## ПРОМПТ 8 — Python-симуляция для диплома (отдельно от веб-приложения)

```
Ты работаешь над проектом SafeAgriRoute. Эта задача НЕ связана с основным 
веб-приложением. Нужно создать автономные Python-скрипты для научной части 
дипломной работы (раздел 1.5 ВКР).

Создать директорию backend/simulation/ со следующими файлами:

ФАЙЛ 1: baseline.py — жадный алгоритм (контроль)

Реализовать простой baseline для сравнения:
- Разбить поле на N равных полос (по горизонтали)
- Назначить по одной полосе каждому дрону
- Маршрут внутри полосы — змейка (boustrophedon)
- НЕ учитывать зоны РЭБ при построении маршрута
- При потере дрона — миссия ОСТАНАВЛИВАЕТСЯ (не перепланирует)
- Метрики: coverage_pct, lost_drones_count, mission_completed: bool

ФАЙЛ 2: runner.py — три сценария

Запускать как: python runner.py

Параметры сцены для всех сценариев:
- Поле: прямоугольник 3km × 1.7km (≈500 га)
- Дроны: 4 шт, battery=100%, speed=10 m/s, autonomy=20 min
- Сетка: шаг 22м → ~700 точек покрытия

СЦЕНАРИЙ 1 — штатный режим:
  Условия: нет зон РЭБ
  Запустить: baseline vs SafeAgriRoute алгоритм
  Метрики: coverage_pct, total_time_sec, mean_IRM, waypoints_count
  Гипотеза: результаты должны быть схожи (алгоритм не хуже в штатном режиме)

СЦЕНАРИЙ 2 — статичные зоны РЭБ:
  Условия: 2 зоны jammer, severity=0.8, каждая 400×300м, 
           расположены в 30% и 70% длины поля
  Запустить: baseline vs SafeAgriRoute
  Метрики: coverage_pct, lost_drones (baseline теряет тех кто летит сквозь зоны),
           mean_IRM, routes_through_jammer_count
  Гипотеза: SafeAgriRoute покрывает >= 90% поля, baseline теряет 1-2 дрона

СЦЕНАРИЙ 3 — динамическое перепланирование:
  Условия: зона jammer появляется на 40% прогресса миссии, 
           дрон #2 выбывает на 60% прогресса
  Запустить: baseline vs SafeAgriRoute
  Метрики: coverage_pct в моменты t=0%, 25%, 50%, 75%, 100% миссии
           (для графика "покрытие во времени")
  Гипотеза: SafeAgriRoute финиширует с > 85% покрытия, baseline < 60%

ФАЙЛ 3: visualizer.py — графики для диплома

Генерировать PNG файлы в директорию simulation/results/:

1. heatmap_scenario2.png — тепловая карта рисков поля со сценария 2
   (matplotlib imshow, colormap='RdYlGn_r', colorbar с подписью "Уровень риска")

2. routes_comparison_scenario2.png — маршруты baseline vs SafeAgriRoute
   Subplot 1x2: слева baseline маршруты (красные линии через зоны), 
   справа SafeAgriRoute (зелёные линии в обход)

3. coverage_timeline_scenario3.png — линейный график
   X: прогресс миссии 0-100%, Y: % покрытия поля
   Две линии: baseline (красная пунктирная) vs SafeAgriRoute (зелёная)
   Вертикальная метка в 40% "Зона РЭБ появилась"
   Вертикальная метка в 60% "Дрон #2 потерян"

4. summary_table.png — итоговая таблица по трём сценариям
   (matplotlib table)

Все графики должны быть в высоком разрешении: dpi=300, figsize=(12, 8)
Подписи осей и заголовки на РУССКОМ языке.

ПРОВЕРЬ СЕБЯ после реализации:
1. Запустить: cd backend && python simulation/runner.py
   Должно выполниться без ошибок за < 60 сек
2. Проверить что в simulation/results/ появились 4 PNG файла
3. Проверить результаты Сценария 1: coverage_pct baseline ≈ coverage_pct SafeAgriRoute
   (разница < 5%) — это доказывает что алгоритм не деградирует в штатном режиме
4. Проверить результаты Сценария 2: 
   SafeAgriRoute coverage > 85%, baseline coverage < 70%
5. Проверить результаты Сценария 3:
   SafeAgriRoute финальное coverage > 80%, baseline < 55%
6. Открыть PNG файлы и убедиться что графики читаемы и подписаны
```

## ПРОМПТ 9 — Слой признаков из телеметрии (Feature extraction)

```
Ты работаешь над проектом SafeAgriRoute. Нужно заложить основу нововведений из ТЗ §10:
из потока данных, который уже доступен через MAVLink (и при необходимости —
расширить типы читаемых сообщений), вычислять числовые признаки «качества
навигации и связи» без ML.

ЗАДАЧА:
Создать модуль backend/app/services/telemetry_features.py (или аналогичное имя).

ФУНКЦИИ (минимум):
1. update_from_mavlink_snapshot(drone_id, messages_dict | dataclass) -> None
   Накопление скользящего окна последних N точек (например N=20) по каждому дрону:
   - lat, lng, alt (из GLOBAL_POSITION_INT или эквивалент)
   - groundspeed, heading
   - battery
   При наличии в сообщениях: fix_type, satellites_visible, eph/epv (если приходят).

2. compute_gnss_stability_score(drone_id) -> float ∈ [0, 1]
   Эвристика: низкий score при резком скачке скорости/позиции между кадрами,
   при падении числа спутников (если есть), при большой невязке «интегрированного»
   смещения vs новая GNSS-точка (упрощённо: сравнить вектор (v*dt) с delta pos).

3. compute_link_stability_score(drone_id) -> float ∈ [0, 1]
   На основе частоты успешных чтений телеметрии, задержек (если есть метки времени),
   опционально RSSI из доступных MAVLink полей.

4. compute_imu_proxy_score(drone_id) -> float ∈ [0, 1]
   Без реального IMU в MVP: прокси через несогласованность ускорения, выведенного
   из последовательности позиций (вторая разность), нормированная по порогу.

5. get_swarm_outlier_score(drone_id, all_drone_positions) -> float ∈ [0, 1]
   Если у дрона отклонение от медианы скорости/направления роя выше порога — score
   ближе к 0 (выброс).

ТРЕБОВАНИЯ:
- Чистые функции + минимальное состояние в памяти (defaultdict + deque).
- Подробные docstring с ссылкой на ТЗ §10.
- Unit-тесты: backend/tests/test_telemetry_features.py с синтетическими рядами
  (резкий скачок позиции → низкий GNSS score).

ПРОВЕРКА:
pytest backend/tests/test_telemetry_features.py — зелёный.
```

## ПРОМПТ 10 — Threat Fusion (правила / взвешенное слияние)

```
Ты работаешь над проектом SafeAgriRoute. На базе Промпта 9 реализуй модуль
backend/app/services/threat_fusion.py.

ВХОД: словарь scores от telemetry_features (gnss, link, imu_proxy, swarm) ∈ [0,1].

ВЫХОД:
- fused_threat_level: float ∈ [0, 1]  (1 = высокая уверенность в угрозе / деградации)
- breakdown: dict с вкладом каждого признака (для отладки и UI)

АЛГОРИТМ (MVP-версия без ML):
- Взвешенная сумма: w_gnss*gnss + w_link*link + w_imu*imu + w_swarm*swarm,
  веса вынести в константы или Settings (сумма весов = 1).
- Опционально: если любой «сырой» признак < порога 0.3 — усилить вклад GNSS.
- Сглаживание: экспоненциальное сглаживание fused по времени (alpha в config),
  чтобы не дёргать перепланирование на одном шумном кадре.

КОНФИГ: backend/app/core/config.py или .env — FUSION_WEIGHTS, FUSION_THRESHOLD,
  FUSION_SMOOTH_ALPHA.

ТЕСТЫ: синтетические входы → ожидаемый порядок fused (хуже GNSS → выше threat).

ПРОВЕРКА:
pytest backend/tests/test_threat_fusion.py
```

## ПРОМПТ 11 — Интеграция fusion → карта риска / перепланирование

```
Ты работаешь над проектом SafeAgriRoute. Свяжи Промпт 10 с существующими
risk_map.py и replanner.py без ломки MVP.

ЗАДАЧА:
1. В mavlink_service.read_telemetry_loop() (или отдельном фоновом таске) после
   обновления кэша телеметрии вызывать цепочку features → fusion.

2. Если fused_threat_level >= FUSION_THRESHOLD в течение M подряд кадров (debounce):
   - Вариант A (проще): сформировать круговую зону радиусом R вокруг текущих
     (lat,lng) дрона с zone_type=jammer и severity=min(1.0, fused*scale),
     передать в ту же логику, что POST .../risk-zones для миссии (вызов
     replan_on_new_risk_zone или общий хелпер).
   - Вариант B (тоньше): локально поднять risk в ячейках risk_grid вокруг точки
     и пересчитать только затронутые дроны (сложнее — согласовать с replanner).

3. Ограничение частоты: не чаще 1 перепланирования на T секунд на миссию
   (конфиг), чтобы не зациклить OR-Tools.

4. Логирование: structured log при срабатывании автоматического триггера.

ТЕСТЫ:
- Интеграционный тест с моками MAVLink: синтетическая «плохая» телеметрия
  → вызов replanner замокан и проверяется, что произошёл ровно один раз.

ПРОВЕРКА:
pytest с маркером integration (или существующий стиль проекта).
```

## Актуальные пути и API

- REST: префикс **`/api/v1/mission/...`** (например `POST /api/v1/mission/plan`, `GET /api/v1/mission/fields`).
- WebSocket: **`/ws/telemetry`** (симуляция по плану в первом сообщении); для живого MAVLink также есть **`/ws/telemetry/{drone_id}`**.
- Фронтенд: `frontend/src/features/MapDashboard/`, `features/MissionControl/`, `components/`.
