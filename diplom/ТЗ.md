## 1. Назначение системы

SafeAgriRoute — веб-платформа централизованного управления группой сельскохозяйственных БПЛА. Оператор через браузер задаёт поле и зоны РЭБ, система строит оптимальные маршруты с учётом рисков, отправляет waypoints дронам через MAVLink и в реальном времени перепланирует миссию при потере дрона или появлении новой угрозы.

**Базовый сценарий MVP:** новая статическая опасность задаётся **явно** — полигон зоны оператором или заранее из БД (сценарии демо A–B).

**Дополнительно (эвристический контур §10, опционально):** после регистрации контекста миссии (`POST /api/v1/mission/{id}/fusion-context`) бэкенд может **автоматически** сформировать **кандидатную** круговую зону jammer вокруг дрона при высокой оценке угрозы по телеметрии MAVLink и вызвать тот же **`replanner`**, что и при ручном добавлении зоны. Это **не** отменяет базовый сценарий: без вызова `fusion-context` и без пороговых срабатываний поведение остаётся как у «классического» MVP.

**Целевая постановка §10 в полном объёме** (отдельные потоки датчиков, SDR, обучаемые модели) описана в разделе 10 ниже и **не** считается полностью реализованной в текущем коде — см. сноску выше.

**Сценарий использования:** оператор в поле или в офисе, браузер на ноутбуке / планшете. Интернет не нужен — система работает в локальной сети (все компоненты на одной машине для MVP).

## 4. Функциональные требования

### 4.1 Управление полями

|ID|Требование|
|---|---|
|F-01|Оператор рисует полигон поля на интерактивной карте (Leaflet Draw)|
|F-02|Поддержка загрузки поля из файла GeoJSON|
|F-03|Поле сохраняется в БД с геометрией POLYGON WGS84|
|F-04|Список полей доступен через `GET /api/v1/mission/fields` (в MVP объединено с роутером миссии)|

### 4.2 Управление зонами РЭБ

|ID|Требование|
|---|---|
|R-01|Оператор рисует зону РЭБ как полигон на карте перед вылетом|
|R-02|Зона имеет атрибуты: `zone_type` (jammer / restricted), `severity` (0.0–1.0)|
|R-03|Во время активной миссии оператор может добавить новую зону — это триггер перепланирования|
|R-04|Зоны отображаются на карте с цветовой индикацией по severity (жёлтый → красный)|

### 4.3 Управление дронами

|ID|Требование|
|---|---|
|D-01|В системе зарегистрированы 4 дрона с параметрами: name, battery_level (0–100%), speed (м/с), autonomy_minutes|
|D-02|Статусы дрона: IDLE / ACTIVE / LOST / RTL / LANDED|
|D-03|Оператор выбирает какие дроны задействовать в миссии|

### 4.4 Планирование миссии (ядро системы)

|ID|Требование|
|---|---|
|P-01|`POST /api/v1/mission/plan` принимает field_id + drone_ids и запускает алгоритм (зоны РЭБ — из БД)|
|P-02|Алгоритм строит дискретную карту рисков R(i,j) с шагом grid_step=0.0002°|
|P-03|Разбиение поля — Risk-Weighted Voronoi: ячейки с высоким риском имеют вес `w = 1/(1-R+ε)`|
|P-04|Назначение зон дронам: жадный алгоритм по battery_level × autonomy|
|P-05|TSP внутри каждой зоны решается через OR-Tools PATH_CHEAPEST_ARC|
|P-06|Рёбра графа через зоны РЭБ получают штраф: `w_edge = dist × (1 + severity × 500)`|
|P-07|Ответ содержит: маршруты RoutePoint[], reliability_index (IRM), estimated_coverage_pct|
|P-08|Время планирования ≤ 5 с для поля 500 га и 4 дронов|

### 4.5 Исполнение миссии через MAVLink SITL

|ID|Требование|
|---|---|
|M-01|После планирования оператор нажимает "Запустить" — бэкенд подключается к 4 SITL-инстансам через pymavlink|
|M-02|Каждый дрон получает список waypoints через MAVLink `MISSION_ITEM_INT`|
|M-03|Дрон выполняет команду `SET_MODE → AUTO` и начинает полёт по маршруту|
|M-04|Бэкенд непрерывно читает телеметрию: `GLOBAL_POSITION_INT`, `BATTERY_STATUS`, `HEARTBEAT`|
|M-05|Телеметрия стримится на фронт через WebSocket каждые 200 мс|
|M-06|Формат WS-сообщения: `{drone_id, lat, lng, alt, battery, status, heading}`|

### 4.6 Динамическое перепланирование

|ID|Требование|
|---|---|
|RE-01|Триггер 1: оператор нажимает "Потерять дрон #N" → статус LOST, запускается replanner|
|RE-02|Триггер 2: оператор добавляет новую зону РЭБ во время миссии → запускается replanner|
|RE-03|Replanner определяет непокрытые waypoints выбывшего дрона|
|RE-04|Непокрытые waypoints распределяются между активными дронами пропорционально residual_capacity|
|RE-05|Для затронутых дронов пересчитывается TSP через OR-Tools|
|RE-06|Новые waypoints отправляются дронам через MAVLink `MISSION_ITEM_INT` (обновление миссии в полёте)|
|RE-07|Фронт получает обновлённые маршруты через WebSocket и перерисовывает карту|
|RE-08|Время перепланирования ≤ 500 мс|

### 6.1 Запуск SITL

```bash

sim_vehicle.py -v ArduCopter --instance 0 --out=tcp:127.0.0.1:14550
sim_vehicle.py -v ArduCopter --instance 1 --out=tcp:127.0.0.1:14560
sim_vehicle.py -v ArduCopter --instance 2 --out=tcp:127.0.0.1:14570
sim_vehicle.py -v ArduCopter --instance 3 --out=tcp:127.0.0.1:14580
```

Или через Docker-образ `ardupilot/ardupilot-sitl`.

### 6.2 Подключение из бэкенда (pymavlink)

```python
from pymavlink import mavutil

drone = mavutil.mavlink_connection('tcp:127.0.0.1:14550')
drone.wait_heartbeat()

# Отправка waypoints
drone.mav.mission_item_int_send(
    target_system=1,
    target_component=1,
    seq=0,
    frame=mavutil.mavlink.MAV_FRAME_GLOBAL_RELATIVE_ALT_INT,
    command=mavutil.mavlink.MAV_CMD_NAV_WAYPOINT,
    current=0, autocontinue=1,
    param1=0, param2=0, param3=0, param4=0,
    x=int(lat * 1e7),
    y=int(lng * 1e7),
    z=30.0  # высота полёта 30 м
)

# Старт миссии
drone.mav.command_long_send(
    1, 1,
    mavutil.mavlink.MAV_CMD_MISSION_START,
    0, 0, 0, 0, 0, 0, 0, 0
)
```

### 6.3 Чтение телеметрии

```python
async def read_telemetry(drone, drone_id: int):
    while True:
        msg = drone.recv_match(
            type=['GLOBAL_POSITION_INT', 'BATTERY_STATUS', 'HEARTBEAT'],
            blocking=False
        )
        if msg:
            yield {
                "drone_id": drone_id,
                "lat": msg.lat / 1e7,
                "lng": msg.lon / 1e7,
                "battery": msg.battery_remaining,
                "status": parse_status(msg)
            }
        await asyncio.sleep(0.2)
```

### 6.4 Обновление миссии в полёте (для replanner)

```python
# Очистить текущую миссию
drone.mav.mission_clear_all_send(1, 1)

# Загрузить новые waypoints
drone.mav.mission_count_send(1, 1, len(new_waypoints))
for i, wp in enumerate(new_waypoints):
    drone.mav.mission_item_int_send(...)

# Продолжить выполнение
drone.mav.command_long_send(
    1, 1,
    mavutil.mavlink.MAV_CMD_MISSION_START, ...
)
```

### 6.5 Симуляция потери дрона

Для демо: бэкенд просто перестаёт получать `HEARTBEAT` от одного SITL-инстанса (или оператор нажимает кнопку → бэкенд игнорирует телеметрию от drone_id и помечает LOST). Физически SITL продолжает работать — нам не важно что дрон "завис", важно что система правильно перепланирует.

## 10. Целевая модель: динамическое определение угрозы по нескольким источникам

Раздел задаёт **научно-техническую перспективу**. **Полная** таблица источников и сигналов ниже **не реализована** в текущем коде целиком.

**Что есть в репозитории (MVP-надстройка):** один поток **MAVLink-телеметрии** (в т.ч. координаты, скорость, батарея, при наличии полей — задел под качество GNSS/канала); эвристики **без бортового IMU** (в т.ч. *IMU-прокси* по рывкам траектории из последовательности позиций); **rule-based fusion**; опционально — **кандидатная** круговая зона jammer и вызов `replanner` (см. сноску к разделу 1 и [`Промпты.md`](Промпты.md) 9–12). Ручной ввод полигонов и сценарии A–B остаются основным демо-путём.

**Цель (перспектива):** повышать локальный риск `R(x,y,t)` или строить **кандидатные** зоны РЭБ, когда заранее нет полигонов угроз, опираясь на несогласованность наблюдений и правила/лёгкие модели на бэкенде.

**Источники и признаки (целевые примеры; в коде — упрощённо, по одному MAVLink-потоку и эвристикам):**

| Источник | Сигналы | Смысл для риска |
|----------|---------|-----------------|
| **GNSS / GPS** | DOP, число спутников, скачки решения, при наличии — индикаторы помех; невязка с предсказанием фильтра | Jamming / spoofing, снижение доверия к позиции |
| **IMU** (акселерометр, гироскоп) | Расхождение интегрированного движения с обновлениями GNSS; рывки при «гладкой» заявленной траектории | Подмена координат, потеря согласованной навигации. *В MVP-коде: прокси по вторым разностям позиции (GNSS-трек), не отдельный датчик* |
| **Барометр** | Несогласованность высоты с профилем миссии при «нормальном» GNSS | Дополнительный признак аномалии |
| **Радиоканал (MAVLink)** | RSSI / качество линка, задержки, потери пакетов | Падение `r_coverage`, экранирование, перегрузка спектра |
| **Согласованность роя** | Выброс одного БПЛА относительно соседей при схожих маршрутах | Локальная деградация ГНСС/связи |
| **Спектр (перспектива)** | SDR: энергия в полосах GNSS L1/L2 | Широкополосная помеха без заранее известной геометрии |
| **Внешние слои** | Ограничения воздушного пространства, при необходимости метео | Уточнение `r_zone` |
| **Оператор** | Полигон + severity | Основной канал в MVP |

**Слияние (fusion):** в целевой постановке — модуль оценки угрозы по сегментам трека (взвешенное голосование, байесовское обновление или пороговые правила) с выдачей в **`risk_map` / replanner** без смены общей логики перепланирования. *В текущем коде:* взвешенная сумма признаков + EMA (`threat_fusion.py`), без байесовского слоя.

## 11. Что НЕ входит в MVP

- Реальный физический дрон (только SITL)
- Поддержка PX4 (только ArduPilot)
- Мобильное приложение
- **Полная** реализация таблицы источников §10 (отдельные драйверы «железного» GNSS/IMU, SDR, обучаемые модели, байесовское fusion уровня научной постановки). При этом **упрощённый** контур по MAVLink с эвристиками и rule-based fusion (опционально авто-кандидатная зона + replan) **входит** в репозиторий — см. раздел 1, §10 и [`Промпты.md`](Промпты.md) 9–12; он **не** считается выполнением этого пункта в полном объёме
- Интеграция с **закрытыми коммерческими** картами РЭБ без открытого API
- 3D-маршрутизация (все маршруты на постоянной высоте 30 м)
- Зарядные станции / смена батарей
- Более 4 одновременных дронов
