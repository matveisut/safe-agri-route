~~## ПРОМПТ 1 — Risk-Weighted Voronoi (замена K-Means)

```
Ты работаешь над проектом SafeAgriRoute. Твоя задача — переработать алгоритм 
разбиения поля в файле backend/app/services/routing_service.py.

ТЕКУЩЕЕ СОСТОЯНИЕ:
Используется K-Means кластеризация, которая делит точки поля только по 
географическому расстоянию, игнорируя карту рисков.

ЧТО НУЖНО СДЕЛАТЬ:
Заменить K-Means на Risk-Weighted Voronoi разбиение по следующей спецификации:

1. Каждая точка поля (из grid с шагом 0.0002°) получает вес:
   weight = 1.0 / (1.0 - risk + 1e-6)
   где risk = значение из risk_grid[i][j] ∈ [0, 1]

2. Разбиение на N зон (N = количество дронов) реализуется через 
   взвешенный Lloyd's algorithm:
   - Инициализировать N центроидов равномерно внутри поля
   - Итеративно (max 50 итераций):
     a. Назначить каждую точку ближайшему центроиду (по евклидову расстоянию)
     b. Сдвинуть каждый центроид в средневзвешенный центр своей зоны:
        new_center = sum(point * weight) / sum(weight)
   - Остановиться при смещении центроидов < 1e-6

3. Назначение дронов на зоны — жадный алгоритм:
   - Вычислить суммарный вес каждой зоны: zone_load = sum(weight) для всех точек зоны
   - Отсортировать зоны по zone_load по убыванию
   - Отсортировать дронов по (battery_level * autonomy_minutes) по убыванию
   - Назначить: самый "сильный" дрон → самая тяжёлая зона

4. Добавить вычисление IRM после построения маршрутов:
   IRM = 1.0 - mean(risk значений всех waypoints маршрута)
   IRM должен возвращаться в ответе /plan как поле reliability_index (float 0-1)

5. Добавить вычисление estimated_coverage_pct:
   estimated_coverage_pct = (количество достижимых точек / общее количество точек поля) * 100

ВАЖНО:
- Сохранить существующий интерфейс функции plan_mission() — не менять сигнатуру
- OR-Tools PATH_CHEAPEST_ARC для TSP внутри каждой зоны оставить без изменений
- Penalty-система для рёбер через зоны РЭБ (severity * 500) оставить без изменений
- Добавить поля reliability_index и estimated_coverage_pct в Pydantic-схему ответа

ПРОВЕРЬ СЕБЯ после реализации:
1. Написать unit-тест в backend/tests/test_routing.py:
   - Создать тестовое поле (простой прямоугольник)
   - Создать зону РЭБ severity=0.9 внутри поля
   - Вызвать plan_mission() с 2 дронами
   - Проверить: ни один waypoint не лежит внутри зоны РЭБ (или их risk < 0.5)
   - Проверить: reliability_index присутствует и ∈ [0, 1]
   - Проверить: estimated_coverage_pct присутствует и > 0
2. Запустить pytest backend/tests/test_routing.py — все тесты должны пройти
3. Вызвать POST /api/v1/missions/plan через curl и убедиться что в ответе есть 
   reliability_index и estimated_coverage_pct
```

~~## ПРОМПТ 2 — Модуль risk_map.py

```
Ты работаешь над проектом SafeAgriRoute. Твоя задача — создать новый файл
backend/app/services/risk_map.py

ЗАДАЧА:
Реализовать функцию build_risk_map() которая строит дискретную карту рисков 
для поля с учётом зон РЭБ.

СПЕЦИФИКАЦИЯ:

def build_risk_map(
    field: Polygon,           # Shapely Polygon — контур поля
    risk_zones: List[dict],   # [{"geometry": Polygon, "severity": float, "zone_type": str}]
    grid_step: float = 0.0002 # шаг сетки в градусах (~22 м)
) -> tuple[np.ndarray, list[tuple], list[tuple]]:
    """
    Возвращает:
    - risk_grid: np.ndarray shape (N, M), значения ∈ [0, 1]
    - grid_points: list[(lat, lng)] — координаты ячеек внутри поля
    - grid_indices: list[(i, j)] — индексы ячеек в risk_grid
    """

АЛГОРИТМ:
1. Получить bounding box поля: minx, miny, maxx, maxy = field.bounds
2. Создать numpy сетку с шагом grid_step
3. Для каждой ячейки (i, j):
   a. Если точка НЕ внутри поля (field.contains(Point(x, y))) → пропустить
   b. Вычислить r_jammer:
      Для каждой зоны типа "jammer":
        dist = Point(x,y).distance(zone.geometry)
        influence_radius = 0.005  # ~500м в градусах
        r_jammer = max(r_jammer, 1.0 - min(dist / influence_radius, 1.0))
      Если точка ВНУТРИ зоны jammer → r_jammer = zone.severity
   c. Вычислить r_zone:
      r_zone = zone.severity если точка внутри зоны типа "restricted", иначе 0.0
   d. Итоговый риск: R(i,j) = min(1.0, 0.5 * r_jammer + 0.35 * r_zone + 0.15 * 0.0)
      (r_coverage = 0 для MVP, НСУ всегда в зоне досягаемости)
4. Нормализовать risk_grid на [0, 1]

ДОПОЛНИТЕЛЬНО:
Добавить функцию get_risk_for_point(risk_grid, grid_meta, lat, lng) -> float
которая возвращает значение риска для произвольной точки по интерполяции.

ПРОВЕРЬ СЕБЯ после реализации:
1. Создать backend/tests/test_risk_map.py:
   - Тест 1: поле без зон → все значения risk_grid == 0.0
   - Тест 2: зона "jammer" severity=1.0 в центре поля → центральные ячейки имеют risk > 0.8
   - Тест 3: зона "restricted" severity=0.9 → ячейки внутри зоны имеют risk > 0.8
   - Тест 4: ячейки вне поля не присутствуют в grid_points
2. Запустить pytest backend/tests/test_risk_map.py — все тесты зелёные
3. Убедиться что routing_service.py теперь вызывает build_risk_map() 
   и передаёт результат в алгоритм Voronoi из Промпта 1
```

~~## ПРОМПТ 3 — replanner.py

```
Ты работаешь над проектом SafeAgriRoute. Твоя задача — создать новый файл
backend/app/services/replanner.py

Этот модуль — главная научная новизна дипломной работы. Он отвечает за 
динамическое перепланирование миссии в двух сценариях.

ЗАВИСИМОСТИ: Промпты 1 и 2 должны быть выполнены. Используй build_risk_map() 
и алгоритм Voronoi из routing_service.py.

СЦЕНАРИЙ A — потеря дрона:

async def replan_on_drone_loss(
    mission_id: int,
    lost_drone_id: int,
    db: AsyncSession
) -> dict:  # возвращает обновлённые маршруты для активных дронов

АЛГОРИТМ:
1. Загрузить миссию из БД, получить все маршруты
2. Найти непокрытые waypoints потерянного дрона:
   uncovered = [wp for wp in lost_drone.route if wp.visited == False]
3. Получить список активных дронов (status == ACTIVE)
4. Если активных дронов 0 → вернуть {"status": "mission_failed"}
5. Вычислить residual_capacity для каждого активного дрона:
   cap(d) = d.battery_remaining * d.speed * d.autonomy_minutes / 100
6. Распределить uncovered waypoints между активными дронами:
   - Нормализовать cap(d): weight(d) = cap(d) / sum(all caps)
   - Назначить каждому дрону долю waypoints пропорционально weight(d)
   - Стараться назначать географически близкие waypoints (сортировка по расстоянию 
     от последней позиции дрона)
7. Для каждого затронутого дрона пересчитать TSP через OR-Tools 
   (объединить его оставшиеся waypoints + новые назначенные)
8. Обновить маршруты в БД
9. Вычислить новый IRM для всей миссии
10. Вернуть {"status": "replanned", "updated_routes": [...], "new_irm": float}

СЦЕНАРИЙ B — новая зона РЭБ:

async def replan_on_new_risk_zone(
    mission_id: int,
    new_zone: dict,  # {"geometry": GeoJSON, "severity": float, "zone_type": str}
    db: AsyncSession
) -> dict:

АЛГОРИТМ:
1. Загрузить миссию и все активные маршруты
2. Инкрементально обновить risk_grid только для bbox новой зоны
3. Для каждого активного дрона:
   a. Получить оставшиеся waypoints (ещё не посещённые)
   b. Проверить пересечение каждого сегмента маршрута с new_zone:
      from shapely.geometry import LineString
      intersects = LineString([wp_a, wp_b]).intersects(new_zone_polygon)
   c. Если есть пересечение → пересчитать TSP для оставшихся waypoints 
      с обновлёнными весами рёбер (новая зона добавляет penalty)
4. Обновить маршруты в БД
5. Вернуть обновлённые маршруты

ТРЕБОВАНИЯ К ПРОИЗВОДИТЕЛЬНОСТИ:
- replan_on_drone_loss должен выполняться ≤ 500 мс для 4 дронов и 200 waypoints
- Использовать asyncio.run_in_executor для CPU-интенсивных операций OR-Tools

ПРОВЕРЬ СЕБЯ после реализации:
1. Создать backend/tests/test_replanner.py:
   - Тест 1 (потеря дрона): создать миссию 4 дронов, пометить 1 как LOST,
     вызвать replan_on_drone_loss(), проверить что uncovered waypoints 
     распределены между 3 оставшимися (суммарное покрытие не уменьшилось)
   - Тест 2 (новая зона): создать миссию, добавить зону РЭБ пересекающую 
     маршрут дрона №1, вызвать replan_on_new_risk_zone(),
     проверить что новый маршрут дрона №1 не пересекает зону
   - Тест 3 (производительность): replan_on_drone_loss() завершается < 500 мс
2. Запустить pytest backend/tests/test_replanner.py
3. Добавить эндпоинт POST /api/v1/missions/{id}/simulate-loss?drone_id=N
   который вызывает replan_on_drone_loss() и возвращает обновлённые маршруты
4. Добавить эндпоинт POST /api/v1/missions/{id}/risk-zones 
   который добавляет зону во время миссии и вызывает replan_on_new_risk_zone()

~~## ПРОМПТ 4 — mavlink_service.py

```
Ты работаешь над проектом SafeAgriRoute. Твоя задача — создать файл
backend/app/services/mavlink_service.py

Этот модуль отвечает за реальное взаимодействие с дронами через протокол MAVLink.
В MVP дроны — это инстансы ArduPilot SITL, но архитектура должна поддерживать 
и реальные дроны без изменения кода.

КОНФИГУРАЦИЯ:
Адреса SITL берутся из переменной окружения SITL_HOSTS:
SITL_HOSTS=tcp:127.0.0.1:14550,tcp:127.0.0.1:14560,tcp:127.0.0.1:14570,tcp:127.0.0.1:14580

Каждый адрес соответствует одному дрону. Дрон с drone_id=1 → первый адрес и т.д.

КЛАСС MAVLinkService:

class MAVLinkService:
    def __init__(self):
        self.connections: dict[int, mavutil.mavlink_connection] = {}
        self.telemetry: dict[int, dict] = {}  # кэш последней телеметрии

    async def connect_all(self) -> None:
        """Подключиться ко всем SITL инстансам при старте бэкенда"""

    async def upload_mission(self, drone_id: int, waypoints: list[dict]) -> bool:
        """
        Загрузить маршрут на дрон через MISSION_ITEM_INT.
        waypoints = [{"lat": float, "lng": float, "alt": float}, ...]
        Возвращает True если миссия загружена успешно (получен MISSION_ACK).
        """

    async def start_mission(self, drone_id: int) -> bool:
        """
        Перевести дрон в AUTO режим и запустить миссию.
        Последовательность команд:
        1. MAV_CMD_DO_SET_MODE → режим GUIDED (arm)
        2. MAV_CMD_COMPONENT_ARM_DISARM → arm (1)
        3. MAV_CMD_NAV_TAKEOFF → взлёт на 30м
        4. MAV_CMD_DO_SET_MODE → режим AUTO
        """

    async def update_mission(self, drone_id: int, new_waypoints: list[dict]) -> bool:
        """
        Обновить маршрут дрона в полёте (для replanner).
        1. MISSION_CLEAR_ALL
        2. Загрузить новые waypoints
        3. MAV_CMD_MISSION_START с current_index = 0
        """

    async def read_telemetry_loop(self, drone_id: int) -> AsyncGenerator[dict, None]:
        """
        Бесконечный генератор телеметрии.
        Читает GLOBAL_POSITION_INT, BATTERY_STATUS, HEARTBEAT.
        Возвращает dict каждые 200мс:
        {
            "drone_id": int,
            "lat": float,      # градусы
            "lng": float,      # градусы  
            "alt": float,      # метры над стартом
            "battery": int,    # процент 0-100
            "heading": int,    # курс 0-360
            "status": str,     # ACTIVE / LOST / LANDED / RTL
            "groundspeed": float
        }
        """

    async def simulate_drone_loss(self, drone_id: int) -> None:
        """
        Для демо: прекратить читать телеметрию от drone_id,
        пометить его статус как LOST в self.telemetry.
        НЕ останавливать SITL физически.
        """

ИНТЕГРАЦИЯ С СУЩЕСТВУЮЩИМ КОДОМ:
- Заменить фейковый цикл в telemetry.py на вызов mavlink_service.read_telemetry_loop()
- При POST /api/v1/missions/{id}/start вызывать:
  1. mavlink_service.upload_mission(drone_id, waypoints) для каждого дрона
  2. mavlink_service.start_mission(drone_id) для каждого дрона
- После replanner: mavlink_service.update_mission(drone_id, new_waypoints)

ОБРАБОТКА ОШИБОК:
- Если SITL недоступен (нет heartbeat за 5 сек) → статус дрона LOST, 
  запустить replanner автоматически
- Все ошибки логировать через logging, не падать

ПРОВЕРЬ СЕБЯ после реализации:

СРЕДА: Windows 10/11. Есть два способа запустить SITL — выбери один:

СПОСОБ А (рекомендуется) — через WSL2:
  Предварительно (один раз):
    wsl --install -d Ubuntu-22.04
    # В WSL2 Ubuntu:
    git clone https://github.com/ArduPilot/ardupilot.git
    cd ardupilot && git submodule update --init --recursive
    Tools/environment_install/install-prereqs-ubuntu.sh -y
    . ~/.profile

  Запуск одного SITL инстанса (в WSL2 терминале):
    cd ardupilot
    sim_vehicle.py -v ArduCopter \
      --custom-location=45.0448,41.9734,0,0 \
      --out=tcp:0.0.0.0:14550 \
      --no-mavproxy

  пробрасывает порты из Ubuntu в Windows.

СПОСОБ Б — через Docker Desktop (проще, но медленнее):
  Предварительно: установить Docker Desktop для Windows с WSL2-бэкендом.

function riskToColor(risk: number): string {
  if (risk < 0.5) {
    // green → yellow
    const r = Math.round(risk * 2 * 255)
    return `rgb(${r}, 255, 0)`
  } else {
    // yellow → red
    const g = Math.round((1 - (risk - 0.5) * 2) * 255)
    return `rgb(255, ${g}, 0)`
  }
}
