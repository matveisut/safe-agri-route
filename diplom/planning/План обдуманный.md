## 1.2 Анализ угроз и формализация метрики риска (10 стр.)

_Мост между курсовой и дипломом. Берёшь готовый материал из курсовой, переформулируешь под новый фокус._

**1.2.1 Специфика угроз в условиях РЭБ-среды (3 стр.)**

- Почему классические модели угроз БПЛА неприменимы напрямую: теоретический хакер vs реальный РЭБ
- Реальный контекст: запреты полётов на юге России 2022–2025, массовое применение систем РЭБ
- Классификация актуальных угроз:
    - УБИ.131 — глушение сигналов ГНСС (jamming): **главная угроза**, высокий риск
    - УБИ.192 — подмена сигналов ГНСС (spoofing): средний риск, академический сценарий
    - УБИ.014 — перехват канала управления MAVLink: средний риск
    - Динамическое закрытие воздушного пространства: организационная угроза с физическими последствиями
- Обоснование выбора УБИ.131 как основного объекта формализации

**1.2.2 Формализация метрики деструктивного воздействия (4 стр.)**

Это ключевое место раздела — математический вклад.

- Постановка задачи: каждой точке пространства над полем присваивается числовой коэффициент риска `r(x,y) ∈ [0, 1]`
- Три компоненты метрики:
    - `r_jammer(x,y)` — вероятность потери ГНСС-сигнала, функция расстояния до известных зон РЭБ
    - `r_zone(x,y)` — бинарный флаг закрытого воздушного пространства
    - `r_coverage(x,y)` — качество радиосвязи с наземной станцией управления
- Итоговая метрика: `R(x,y) = α·r_jammer + β·r_zone + γ·r_coverage`, где α+β+γ=1
- Калибровка весов α, β, γ на основе матрицы рисков из курсовой
- Формирование дискретной карты рисков: поле разбивается на grid, каждой ячейке присваивается `R(i,j)`
- Графическое представление: тепловая карта рисков поверх полигона поля (рисунок)

**1.2.3 Сценарии воздействия для экспериментальной части (3 стр.)**

- Адаптация модели нарушителя из курсовой под реальный контекст
- Три формализованных сценария:
    - С1: зоны РЭБ статичны, известны до вылета
    - С2: зона РЭБ появляется динамически во время миссии (**в MVP** — ввод полигона оператором; **в целевой модели** — также кандидатная зона из телеметрии: невязки GNSS/IMU, падение RSSI, выброс из «кластера» роя)
    - С3: зона РЭБ + потеря одного дрона одновременно
- Формальное описание каждого сценария как входных параметров алгоритма; для С2 отдельно описать **канал слияния признаков** (см. `ТЗ.md` §10)

### 1.4.1 Постановка задачи и обзор существующих подходов (3 стр.)

**Формальная постановка:**

- Входные данные: полигон поля F, множество дронов D = {d₁...dₙ} с характеристиками (battery, speed, autonomy), дискретная карта рисков R(i,j) ∈ [0,1]
- Выходные данные: набор маршрутов {path₁...pathₙ}, индекс надёжности миссии IRM ∈ [0,1]
- Критерий оптимальности: минимизация суммарного взвешенного риска вдоль маршрутов при максимизации процента покрытия поля

**Таксономия существующих подходов (таблица сравнения):**

По обзору 88 исследований 2020–2025 гг. (Yazid et al., SSRN 2026) алгоритмы делятся на четыре класса:

|Класс|Примеры|Применимость к задаче|Ограничение|
|---|---|---|---|
|Классические|A*, Dijkstra, D*|Поиск пути в графе с весами риска|Не масштабируются на рой, нет учёта покрытия|
|Метаэвристические|GA, PSO, ACO, GWO|Глобальная оптимизация сложных функций|Медленная сходимость, сложно гарантировать real-time|
|Декомпозиция + CVRP|Voronoi + OR-Tools|Разбиение поля + оптимальные маршруты внутри зон|Стандартный Voronoi не учитывает риски|
|Обучение с подкреплением|Q-learning, DRL|Адаптация в динамической среде|Требует обучения, не интерпретируемо|

**Вывод:** ни один существующий подход не решает одновременно задачу risk-aware декомпозиции поля и динамического перепланирования при потере агента. Предлагается гибридный алгоритм: **Compensated Voronoi Decomposition + weighted CVRP (OR-Tools) + event-driven replanner**.

### 1.4.2 Алгоритм построения карты рисков `risk_map` (2 стр.)

**Реализация:** `backend/app/services/risk_map.py`

```
Вход: field: Polygon, risk_zones: List[RiskZone], grid_step: float
Выход: risk_grid: np.ndarray[float], shape=(N, M), значения ∈ [0,1]
```

**Алгоритм:**

1. Разбить bounding box поля на сетку с шагом `grid_step` (≈0.0002° ≈ 22 м)
2. Для каждой ячейки (i,j) внутри полигона поля вычислить три компоненты:
    - `r_jammer = 1 - min(dist(cell, zone) / influence_radius, 1.0)` для зон типа `jammer`
    - `r_zone = 1.0` если ячейка внутри зоны типа `restricted`, иначе `0.0`
    - `r_coverage = max(0, 1 - dist(cell, НСУ) / max_comm_range)`
3. Агрегация: в **рабочем коде** jammer и restricted комбинируются иначе, чем в учебной формуле с α, β, γ: см. `safe-agri-route/backend/app/services/risk_map.py` и `knowledge/algorithms.md` (зона jammer с затуханием по расстоянию, restricted — по вхождению в полигон; затем нормализация). В тексте диплома можно оставить α, β, γ как **целевую модель**, явно указав, что MVP-реализация упрощает вклад `r_coverage` (НСУ не моделируется отдельно).
4. Нормализация на [0, 1]

**Визуализация:** тепловая карта рисков поверх полигона поля (рисунок).  
**Сложность:** O(N·M·K), где N×M — размер сетки, K — число зон риска.

### 1.4.3 Алгоритм декомпозиции поля: Compensated Voronoi (4 стр.)

**Обоснование выбора метода:**

Стандартный Voronoi делит пространство на зоны равноудалённости от центроидов — он применяется в задачах покрытия роем (Luna et al., Journal of Field Robotics 2024; Zhao et al., IEEE 2024). Однако он не учитывает неоднородность среды. Для враждебных сред предложен **Compensated Voronoi Diagram** (Shin et al., Int. J. Aeronautical and Space Sciences 2024) — зоны Вороного строятся с учётом весовых коэффициентов угроз.

**Адаптация для SafeAgriRoute — Risk-Weighted Voronoi:**

```python
# Стандартный Voronoi (было в K-means):
weight = 1.0  # все точки одинаковы

# Risk-Weighted Voronoi (предлагается):
risk = risk_grid[i][j]
point_weight = 1.0 / (1.0 - risk + 1e-6)
# Рискованные точки «тяжелее» → дрон с большей автономностью
# получает зону с бо́льшим суммарным весом, естественно принимая риск
```

**Полный пайплайн `route_planner`:**

1. **Генерация точек покрытия:** сетка точек внутри полигона, исключая ячейки с `R(i,j) = 1.0` (полностью запрещённые зоны)
2. **Weighted Voronoi decomposition:** инициализировать центроиды случайно, итеративно смещать каждый центроид в средневзвешенный центр своей зоны — аналог Lloyd relaxation с весами риска
3. **Назначение дронов на зоны:** жадный алгоритм — дрон с максимальным `battery_level` получает зону с максимальным суммарным весом `Σ w(i,j)`
4. **Построение графа внутри зоны:** NetworkX, вес ребра между точками A и B:
    
    ```
    edge_weight(A,B) = dist(A,B) × (1 + severity × PENALTY_COEFF)
    ```
    
    где `severity` — максимальный риск вдоль отрезка AB (проверяется через `shapely`)
5. **Решение TSP внутри зоны:** OR-Tools `PATH_CHEAPEST_ARC` эвристика
6. **Вычисление IRM:** `IRM = 1 - mean(R(i,j) вдоль маршрута)`

**Сравнение с базовым K-means (ранний прототип / литература):**

|Критерий|K-means (базовый вариант)|Risk-Weighted Voronoi (**текущая реализация в `routing_service.py`**)|
|---|---|---|
|Учёт рисков при разбиении|❌ Нет|✅ Да|
|Дрон может попасть в опасную зону|✅ Да|❌ Маловероятно|
|Учёт гетерогенности дронов|Частично (по battery)|✅ Полный (battery × risk load)|
|Сложность|O(K·N·iter)|O(K·N·iter) — та же|

**Блок-схема алгоритма (рисунок), псевдокод (листинг).**

### 1.4.4 Алгоритм динамического перепланирования `replanner` (4 стр.)

**Обоснование актуальности:**

По обзору Luna et al. (2024) большинство существующих CPP-систем работают только в offline-режиме (75% из 88 исследований). Онлайн-перепланирование при отказе агента — открытая исследовательская задача. Предложенный подход использует event-driven архитектуру, аналогичную RHTA (Hong et al., 2022), но расширенную на сценарий появления динамических зон угроз.

**Реализация:** `backend/app/services/replanner.py`

**Триггер 1: потеря дрона (`drone_status → LOST`)**

```
replan_on_drone_loss(mission, lost_drone_id):
  1. uncovered = [точки маршрута lost_drone, которые ещё не пройдены]
  2. active_drones = [d for d in mission.drones if d.status == ACTIVE]
  3. Для каждого active_drone вычислить residual_capacity:
       cap(d) = d.battery_remaining × d.speed × autonomy_factor
  4. Распределить uncovered между active_drones пропорционально cap(d)
       (задача Bin Packing с взвешенными весами точек)
  5. Для каждого затронутого дрона: пересчитать TSP через OR-Tools
  6. Обновить IRM для всей миссии
  7. Отправить новые маршруты через WebSocket
```

**Триггер 2: появление новой зоны РЭБ**

```
replan_on_new_risk_zone(mission, new_zone):
  1. Пересчитать risk_grid только для ячеек в bounding box new_zone — O(bbox)
  2. Для каждого active_drone:
       intersecting_segments = [сегменты маршрута, пересекающие new_zone]
       if intersecting_segments:
         rebuild waypoints для этого дрона с обновлёнными весами
         re-solve TSP через OR-Tools для изменённого подграфа
  3. Обновить IRM
  4. Отправить обновления через WebSocket
```

**Временна́я сложность перепланирования:**

- Полное перепланирование: O(K·TSP) ≈ O(K·N²) — неприемлемо для real-time
- Инкрементальное (только затронутые дроны): O(k·n²), где k ≤ K, n ≪ N
- На практике для поля 500 га, 4 дронов, grid 22м: k=1–2, n≈50–100 точек → < 200 мс ✅

**Алгоритм A* для локального обхода (опционально):**

Если перестроить маршрут через OR-Tools слишком долго (n > 200 точек), применяется локальный A* только для сегментов, пересекающих новую зону — это снижает время до < 50 мс. Веса ребер в A* включают компоненту риска: `f(n) = g(n) + h(n) + λ·R(n)`, где λ — коэффициент пессимизма.

### 1.4.5 Обзор альтернативных подходов и обоснование выбора (1 стр.)

Здесь показываешь комиссии что ты знаешь литературу и сделал осознанный выбор:

|Подход|Источник|Почему не выбран|
|---|---|---|
|Генетический алгоритм (GA)|Puente-Castro et al., 2024 (arxiv)|Медленная сходимость, нет гарантии real-time для онлайн-перепланирования|
|PSO + ACO (FA3ACO)|Wu et al., Scientific Reports 2025|Хорошо для статичных сред, плохо адаптируется к динамическим угрозам|
|Grey Wolf Optimizer (GWO)|Springer IJCIS 2025|Требует настройки гиперпараметров, сложно интегрировать с OR-Tools|
|Q-learning на Voronoi-графе|Su et al., Aerospace Engineering 2024|Требует предобучения, не интерпретируемо, избыточно для CPP|
|Hybrid PLO|Springer 2025|Показывает лучшие результаты на бенчмарках CEC2017, но нет реализации для Python/OR-Tools|
|**Risk-Weighted Voronoi + OR-Tools**|**Предлагается**|**Детерминированный, интерпретируемый, real-time (< 200 мс), Python-реализация**|

**Вывод:** для задачи покрытия сельскохозяйственного поля с известными зонами риска и требованием real-time перепланирования оптимален детерминированный гибридный подход: декомпозиция через взвешенный Voronoi + CVRP через OR-Tools + инкрементальный replanner. Метаэвристики (GA, PSO, GWO) дают лучшее качество решения в статичных задачах, но проигрывают по времени отклика в динамической среде.

### 1.4.6 Интеграция алгоритма в платформу (2 стр.)

- Точка входа: `POST /api/v1/mission/plan` → `RoutingService.plan_mission()` → ответ с маршрутами + IRM (в коде для ускорения на больших полях используется `step_deg=0.002` в вызове из роутера — отличие от дефолта `0.0002°` в самом `build_risk_map`; это нужно описать в дипломе как компромисс скорость/детализация)
- Асинхронная архитектура FastAPI: планирование выполняется в `asyncio.run_in_executor` — не блокирует event loop
- WebSocket стриминг: после запуска миссии `replanner` подписывается на события `DRONE_LOST` и `NEW_RISK_ZONE`
- Формат ответа `/plan`:
    
    ```json
    {  "mission_id": 42,  "reliability_index": 0.87,  "estimated_coverage_pct": 94.3,  "routes": [    {"drone_id": 1, "waypoints": [{"lat": ..., "lng": ..., "risk": 0.12}, ...]},    ...  ]}
    ```
